## Simple Shell

### Data Structures

The shell uses a local 128-byte character buffer (char line[128]) and an integer index variable to keep track of the current buffer position.

### Algorithms

If no command-line arguments are provided to the kernel, main() enters an interactive loop. Inside the loop, it prints the prompt CSE134> . It then reads keyboard input using input_getc() in a nested loop. Each key is echoed back to the user and stored in the buffer until a newline (\r or \n) is encountered, at which point the buffer is null-terminated. The shell then evaluates the string using strcmp. Known commands like whoami and exit are handled immediately, and "Invalid command" is printed for anything else.

### Synchronization

The shell runs as a single thread, so no explicit synchronization is required. input_getc() safely manages the keyboard hardware buffer and blocks the shell thread while waiting for input, which is a natural form of synchronization with the hardware.

### Design Justification

Reading one character at a time with input_getc() is the correct approach because it blocks the thread until input is available, making the shell responsive without using CPU-heavy polling loops. A fixed-size buffer with a clear bounds check (i < 127) ensures the shell is robust and avoids memory allocation overhead.

## Alarm Clock

### Data Structures

I added a wake_up_ticks member of type int64_t to struct thread (in thread.h) to store the absolute tick count at which a thread should wake up. I also added a global sleep_list in thread.c, initialized in thread_init(), to manage threads that are currently blocked in the THREAD_BLOCKED state. Each sleeping thread uses its existing list_elem to reside on this list.

### Algorithms

When timer_sleep is called, it calculates the absolute wake-up time (timer_ticks() + ticks) and passes this to thread_sleep. thread_sleep marks the thread as blocked, records its wake-up tick, and adds it to the sleep_list. The timer interrupt handler calls thread_awake every tick. thread_awake iterates through the sleep_list; if a thread's wake_up_ticks is less than or equal to the current tick count, thread_unblock is called to wake it.

### Synchronization

The sleep_list is shared between thread_sleep (thread context) and thread_awake (interrupt context), creating a race condition. If an interrupt occurs while thread_sleep is adding a thread to the list but before it calls thread_block(), the thread could be unblocked prematurely. To prevent this, thread_sleep disables interrupts before modifying the list and blocks the thread atomically.

### Design Justification

Blocking threads in a sleep_list is more efficient than busy-waiting because it allows the scheduler to give the CPU to other threads instead of spinning in a loop. Using an absolute tick count rather than a relative count simplifies the comparison in thread_awake to a single check per thread.
